# Target Shooter Rush

Mobile portrait shooting arcade generated by `dev_game/generator` from the
`target-shooter-rush` spec, then productionized with custom gameplay code and
per-game AI image assets.

## Run

```bash
npm install
npm run dev
npm run build
npm run preview
```

## Game Loop

- Tap the moving target before the 45-second timer expires.
- Normal hits award `100`, perfect center hits award `180`, and combo adds `18 * (combo - 1)`.
- Hits add `+1.2s` up to the 45-second cap; misses apply `-30 score`, `-3s`, and combo reset.
- Stage goals advance DAY -> NIGHT -> RUSH with score/time rewards.
- Difficulty rises from elapsed time, hits, and stage bonus, so targets keep getting faster and smaller.
- Best score is persisted in localStorage.

## Production Notes

- Core loop lives in `src/game/scenes/GameScene.js`.
- Target movement lives in `src/game/entities/MovingTarget.js` and `src/game/systems/TargetSystem.js`.
- Tuning lives in `src/game/config/shootingConfig.js`.
- HUD and layout QA hooks live in `src/game/ui/HudUI.js` and `src/game/systems/LayoutRegistry.js`.
- Production image assets live in `assets/images/production/`.
- The gameplay reticle is generated at runtime as `reticle_ui`; the generated `crosshair.png` remains an asset artifact, not the active in-game reticle.
- `player_blaster.png` is loaded for the per-game asset contract, but `GameScene` no longer draws a separate player image over the baked cannon. Runtime uses a small muzzle-flash anchor instead.

## Verified Gates

```bash
npm --prefix dev_game run factory:production-gate -- \
  --project dev_game/generated/target-shooter-rush \
  --require-gpt-imagegen \
  --viewports 390x844,430x932,1080x1920
```

The aggregate gate runs Foundation QA, production-demo QA, image-quality QA,
visual-layout QA, scene-composite QA, and scene-composite pixel inspection.

The game also exposes `window.__TARGET_SHOOTER_QA__()` in the Game scene for a
browser gameplay smoke that fires at the current target center.
