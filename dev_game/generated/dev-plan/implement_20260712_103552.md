## 핵심 결정 기록

### D1. schema v1은 보존하고 v2를 별도 도입한다

- `game-spec.schema.json`을 즉시 파괴적으로 수정하지 않는다.
- `schemaVersion`으로 v1/v2 validator를 선택한다.
- v1 예제는 수정 없이 계속 통과해야 한다.
- 신규 custom-loop 게임은 v2만 사용한다.

### D2. custom-loop CLI는 gameplay를 자동 생성하지 않고 안전한 shell만 생성한다

- `arcade-vertical`은 기존 Spawner/ScoreManager/Foundation을 유지한다.
- `custom-shell`은 Boot/Loading/Home/Game/Pause/GameOver, AudioManager, SaveData, LayoutRegistry, QA hooks만 생성한다.
- custom gameplay 구현 전에는 `implementationStatus: "foundation"`으로 기록하고 Production Demo 완료를 금지한다.

### D3. 규칙 계약은 runtime config를 단일 원본으로 삼는다

- Firebreak의 실제 비용과 시간은 `FIREBREAK_CONFIG`가 소유한다.
- `GAME_RULES`는 해당 config에서 파생한다.
- Home/Tutorial/MissionCoach/HUD는 숫자를 직접 쓰지 않는다.
- QA는 `window.__GAME_RULES__`와 화면 snapshot을 비교한다.

### D4. Capture Matrix는 데이터, 상태 진입은 project adapter가 담당한다

- JSON에 임의 JavaScript를 넣지 않는다.
- `qa/capture-matrix.json`은 상태 이름·expected scene·required IDs·assertions만 선언한다.
- 프로젝트는 `window.__GAME_QA__.gotoState(id)`와 `getState()`를 제공한다.
- 공통 runner는 adapter를 호출하고 screenshots/report/contact sheet를 같은 실행에서 만든다.

### D5. Gate 도입은 warn → required 두 단계로 진행한다

- 기존 v1 프로젝트: 신규 게이트 미구성 시 warning, 기존 Gate 동작 유지.
- schema v2/custom-loop: clarity/capture/input/persistence adapter 미구성 시 실패.
- 안정화 후 신규 생성 게임 전체에 required를 적용한다.

### D6. 외부 이미지/QA 라이브러리를 새로 추가하지 않는다

- contact sheet는 Playwright HTML/CSS grid 캡처로 생성한다.
- 기존 Node 표준 라이브러리와 Playwright를 우선 사용한다.

## Phase 1. Versioned Game Spec과 Custom Shell

### 목표

- arcade Foundation과 custom-loop를 같은 허위 필드 없이 표현한다.

### 구현 태스크

- [x] 현재 schema를 `game-spec.v1.schema.json`으로 보존한다.
- [x] `game-spec.v2.schema.json`에 common/game/canvas/audio/performance를 정의한다.
- [x] v2에 `buildDecision`, `implementationStatus`, `runtimeConfig`, `rules`, `requiredAssetRoles`, `captureMatrix`를 정의한다.
- [x] `cli.mjs`가 `schemaVersion`에 맞는 schema를 선택하도록 변경한다.
- [x] validator에 v2에서 필요한 array `items`, `minItems`, `additionalProperties` 검사만 최소 추가한다.
