## 개발 목적

- 세 게임의 리뷰에서 확인된 결함과 출시 리스크를 게임별로 독립 수정한다.
- 한 에이전트가 다른 게임의 파일이나 공통 Factory 파일을 건드리지 않도록 소유권을 분리한다.
- 각 게임의 수정·재캡처·회귀 검증을 독립적으로 완료한 뒤 통합 Gate에서 한 번에 확인한다.
- 새 기능 추가가 아니라 현재 Vertical Slice/production-demo의 명확성, 검증 재현성, 실기기 준비도를 높인다.

- Phase A/B/C는 서로 독립적으로 진행하며, 각 에이전트는 자기 게임 디렉터리만 수정한다.

- 공통 파일 수정이 필요하면 직접 수정하지 말고 Root agent에 충돌 요청을 남긴다.
- 각 수정은 사용자 증상 또는 QA 증거 → 결함 분류 → 최소 수정 → 동일 조건 before/after 재캡처 순서를 지킨다.
- 한 Phase의 자체 테스트가 실패하면 다음 Phase로 이동하지 않고 같은 Workstream 안에서 원인을 해결한다.

- 기존 기능·규칙·점수·승패 조건을 임의로 변경하지 않는다.
- 기존 Production Gate가 녹색이어도 이번 수정 표면의 재캡처와 상태 샘플을 반드시 남긴다.

게임별 dev 서버와 QA 기본 URL은 다음 값을 고정한다. QA 실행 시에는 항상 해당 게임의 `GAME_QA_URL`을 우선하며, 다른 게임의 포트로 fallback하지 않는다.

| 게임 | 고정 포트 | 기본 QA URL |
|---|---:|---|
| Skybreak Gunship | `5187` | `http://127.0.0.1:5187` |
| Firebreak Commander | `5188` | `http://127.0.0.1:5188` |
| Harbor Crane Commander | `5189` | `http://127.0.0.1:5189` |

- 포트 변경이 필요하면 Root agent가 먼저 이 표와 해당 게임의 `package.json`/QA fallback/문서를 함께 갱신한다.
- `5173`, `5186` 등 임의의 fallback은 이번 계획에서 허용하지 않는다(역사적 리포트 인용은 예외).
- 병렬 실행 예: `GAME_QA_URL=http://127.0.0.1:5187 npm run test:combat`.

## Phase 4. 병렬 결과 통합 및 전체 Gate — Root agent

### 목표

세 에이전트의 독립 수정이 서로 영향을 주지 않았는지 확인하고, 최종 산출물을 통합한다.

### 통합 순서

1. [x] Agent A/B/C의 자체 테스트 결과와 변경 파일 목록을 수집한다.
2. [x] 각 게임 디렉터리 외 변경이 없는지 `git diff --stat`/경로 검사를 수행한다.
3. [x] 각 게임별 Build와 개별 QA를 다시 실행한다.
4. [ ] 세 게임의 Production Gate를 순차 실행한다.
5. [x] 공통 계약 위반·문서 누락·QA evidence 경로를 확인한다.
6. [x] 충돌이 발견되면 해당 에이전트만 재작업하고 다른 Workstream은 재실행하지 않는다.

### 병합/롤백 규칙

- 병합 순서는 Agent A → Agent B → Agent C가 아니라 **A/B/C 자체 검증 완료 후 Root가 논리적으로 순차 Gate**를 실행하는 방식으로 고정한다. 게임 디렉터리가 분리되어 있어 소스 병합 충돌은 발생하지 않아야 한다.
- 각 에이전트는 변경 전 `git diff -- <owned paths>`와 변경 후 동일 명령의 경로 목록을 남긴다. 소유 범위 밖 변경은 즉시 되돌리고 해당 카드만 재검증한다.
- 한 게임의 회귀 실패는 해당 게임 카드만 롤백/재작업한다. 다른 두 게임의 코드는 되돌리지 않는다.
- 통합 실패 시 마지막 PASS evidence와 각 게임의 `dist`를 보존하고, 실패 원인이 확인된 카드에만 수정 커밋을 추가한다. `dist` 수동 패치는 허용하지 않는다.
